01 — masalah

Sebelum Git, begini rasanya "kolaborasi"

proposal_final.docx
proposal_final_v2.docx
proposal_final_v2_REVISI.docx
proposal_final_v2_REVISI_ok.docx
proposal_final_FIX_pakaiini.docx

Tanpa version control: tidak ada riwayat siapa mengubah apa, takut menimpa kerja orang lain, dan tidak ada jalan aman untuk kembali ke versi sebelumnya.

Git lahir tahun 2005, ditulis Linus Torvalds untuk mengelola kode kernel Linux — ribuan kontributor, tanpa satu pun server pusat yang jadi titik gagal. Sekarang jadi standar version control di hampir semua tim software.

02 — arsitektur

Kenapa "distributed" itu penting

Distributed (Git)

  • Setiap clone = salinan lengkap riwayat proyek
  • Commit, lihat log, buat branch — semua tanpa internet
  • Server remote cuma titik sinkronisasi, bukan satu-satunya sumber kebenaran
  • Kalau server remote hilang, riwayat masih hidup di laptop siapa pun

Centralized (SVN, CVS)

  • Riwayat lengkap cuma ada di satu server pusat
  • Butuh koneksi ke server untuk hampir semua operasi
  • Server down atau rusak → riwayat proyek ikut terancam
  • Branching relatif berat & jarang dipakai bebas
03 — konsep inti

Tiga area tempat file kamu "hidup"

Working Directory

Folder proyek asli. Tempat kamu mengedit file sehari-hari.

git add

Staging Area

Ruang tunggu. Daftar perubahan yang sudah kamu tandai untuk disimpan.

git commit

Repository (.git)

Riwayat permanen. Snapshot tersimpan selamanya dengan ID unik.

Ketiganya berurutan: git add memindahkan perubahan dari working directory ke staging, git commit mengunci isi staging jadi snapshot permanen di repository.

05 — melihat perubahan

Membaca perubahan sebelum di-commit

function sapa(nama) {
- console.log("Hai " + nama);
+ console.log(`Halo, ${nama}!`);
+ return true;
}

git diff menampilkan perubahan di working directory yang belum di-stage. Baris merah (-) dihapus, baris hijau (+) ditambahkan.

git diff --staged menampilkan perubahan yang sudah masuk staging area — inilah isi yang akan dibekukan saat kamu menjalankan git commit.

06 — cabang itu murah

Branch cuma sebuah pointer

Di banyak sistem lama, membuat cabang berarti menyalin seluruh file. Di Git, sebuah branch hanyalah penunjuk 40-karakter ke satu commit tertentu — nyaris tanpa biaya untuk dibuat atau dipindah.

HEAD adalah penunjuk ke branch yang sedang aktif. Berpindah branch berarti memindahkan HEAD, bukan menyalin folder.

HEAD → main → commit 7c1a3f2
↳ commit 4e0b881
↳ commit 9d2c7a0

Membuat branch baru = menulis satu baris teks berisi hash commit. Itu saja.

07 — praktik branching

Membuat & berpindah branch

zsh
git branch fitur-login
 
git switch fitur-login
Switched to branch 'fitur-login'
# atau sekaligus buat + pindah:
git switch -c fitur-checkout
Switched to a new branch 'fitur-checkout'
git branch
fitur-checkout
fitur-login
* main

git switch dan git restore (sejak Git 2.23) menggantikan sebagian peran git checkout lama supaya tiap perintah punya satu tugas yang jelas: switch untuk pindah branch, restore untuk membatalkan perubahan file.

08 — riwayat bercabang

Begini bentuknya riwayat yang bercabang

Titik oranye adalah tempat fitur-login bercabang dari main. Sejak itu, dua riwayat berjalan sendiri-sendiri sampai nanti disatukan lewat merge.

09 — menyatukan cabang

Dua cara Git menyatukan branch

Fast-forward

Kalau main tidak punya commit baru sejak branch dibuat, Git cukup menggeser pointer — tidak ada commit gabungan baru.

Three-way merge

Kalau kedua branch sudah berjalan sendiri, Git membuat commit baru beranak dua parent yang menggabungkan keduanya.

zsh
git switch main
git merge fitur-login
10 — dua filosofi

Merge vs. rebase: bukan benar-salah, tapi konteks

git merge

  • Riwayat asli dipertahankan apa adanya
  • Aman dipakai di branch yang sudah dibagikan ke orang lain
  • Menghasilkan satu merge commit tambahan
  • Graf riwayat jadi bercabang — jujur, tapi kadang ramai

git rebase

  • Menulis ulang commit branch di atas commit terbaru main
  • Hasil akhir: riwayat lurus, mudah dibaca
  • Mengubah hash commit — jangan rebase branch yang sudah di-push & dipakai orang lain
  • Cocok untuk merapikan branch pribadi sebelum dibuka jadi pull request
11 — saat git butuh bantuan

Membaca & menyelesaikan konflik

function sapa(nama) {
<<<<<<< HEAD
return "Halo, " + nama;
=======
return `Hai ${nama}!`;
>>>>>>> fitur-login
}
1

Buka file yang bentrok

Git menandai bagian yang bertabrakan dengan <<<<<<<, =======, >>>>>>>.

2

Pilih atau gabungkan kode

Edit manual jadi versi final, lalu hapus semua baris penanda.

3

Tandai selesai

git add nama-file, lalu git commit (atau git merge --continue).

12 — bekerja dengan server

Repo lokal, ketemu repo remote

Repo lokal

Laptop kamu. Riwayat lengkap, bisa dipakai offline.

push → ← pull

origin (remote)

Server bersama — GitHub/GitLab. Titik sinkronisasi tim.

zsh
git clone https://github.com/tim/proyek.git
git remote -v
origin https://github.com/tim/proyek.git (fetch)
git fetch origin
# unduh riwayat terbaru tanpa mengubah file kerja
git pull origin main
# setara: fetch + merge sekaligus
git push origin fitur-login
13 — alur kolaborasi tim

Alur kerja yang dipakai kebanyakan tim

1

Branch dari main

git switch -c fitur/nama-singkat — mulai dari kondisi terbaru.

2

Commit kecil & sering

Tiap commit satu perubahan logis, dengan pesan yang jelas.

3

Push & buka Pull Request

git push origin fitur/nama-singkat, lalu ajukan review ke tim.

4

Review & diskusi

Rekan tim memberi komentar; kamu commit perbaikan di branch yang sama.

5

Merge ke main

Setelah disetujui, gabungkan lewat platform (biasanya squash atau merge commit).

6

Hapus branch fitur

Selesai tugasnya — git branch -d fitur/nama-singkat menjaga repo tetap rapi.

14 — membatalkan dengan aman

Kotak perkakas untuk membatalkan

PerintahMenyentuh areaKapan dipakai
git restore <file>Working directoryBaru edit, mau balik ke versi commit terakhir.
git reset --soft HEAD~1Repository sajaBatalkan commit terakhir, tapi simpan perubahan di staging.
git reset --mixed HEAD~1Repository + stagingBatalkan commit & un-stage, file tetap ada di working directory.
git reset --hard HEAD~1Semua areaBuang permanen — hanya kalau benar-benar yakin.
git revert <hash>Repository (commit baru)Branch sudah dibagikan; butuh cara aman membalik efek commit publik.
git stashWorking directory (sementara)Perlu pindah branch cepat tanpa commit yang belum selesai.
15 — kebersihan repo

Bilang ke Git: "jangan lacak yang ini"

.gitignore
node_modules/
dist/
.env
.DS_Store
*.log

Tiga kategori yang biasanya masuk .gitignore: rahasia (kunci API, .env), hasil build (dist/, .next/), dan dependency besar yang bisa diunduh ulang (node_modules/).

File yang sudah kadung ter-commit tidak otomatis berhenti dilacak hanya dengan menambah pola ke .gitignore — perlu git rm --cached <file> sekali untuk melepaskannya.

16 — kebiasaan baik

Kebiasaan yang bikin repo enak diajak kerja

format pesan commit

type(scope): pesan singkat

feat fitur baru
fix perbaikan bug
refactor ubah struktur, perilaku sama
docs / chore / test sisanya

Atomic commit — satu commit mewakili satu perubahan logis. Memudahkan review, dan memudahkan git revert kalau ada yang salah.

Nama branch yang jelas — fitur/checkout-cepat, fix/login-timeout — langsung kelihatan tujuannya tanpa buka kode.

Tulis pesan untuk dirimu 6 bulan lagi — "fix bug" tidak membantu; "fix: cegah double submit saat checkout" membantu.

17 — referensi cepat

Cheatsheet perintah yang paling sering dipakai

Setup

  • git initmulai repo baru
  • git clone <url>salin repo remote
  • git config --global user.name "Nama"identitas commit

Snapshot

  • git statusapa yang berubah
  • git add .stage semua perubahan
  • git commit -m "…"simpan snapshot
  • git log --oneline --graphriwayat ringkas

Branch

  • git switch -c <nama>buat + pindah branch
  • git switch <nama>pindah branch
  • git merge <nama>gabungkan branch
  • git branch -d <nama>hapus branch selesai

Remote

  • git remote -vlihat remote terhubung
  • git fetch originunduh tanpa merge
  • git pull origin mainunduh + merge
  • git push origin <branch>unggah commit

Undo

  • git restore <file>buang perubahan lokal
  • git reset --soft HEAD~1batalkan commit terakhir
  • git revert <hash>balik efek commit lama
  • git stashsimpan sementara
