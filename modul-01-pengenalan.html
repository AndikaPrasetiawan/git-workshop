Pengenalan Git & Version Control

Sebelum mengetik satu perintah pun, penting untuk paham masalah apa yang sebenarnya diselesaikan oleh Git.

Di modul ini
  1. Apa itu Version Control System
  2. Kenapa memakai Git
  3. Git vs GitHub
  4. Konsep dasar & alur kerja

Apa itu Version Control System (VCS)?

Bayangkan kalian sedang menulis kode dan menyimpan file dengan nama tugas_final.js, tugas_final_v2.js, tugas_final_v2_FIX.js, tugas_final_v2_FIX_beneran.js. Ini adalah cara manual (dan menyakitkan) untuk melacak perubahan. Version Control System menggantikan kekacauan itu dengan sistem yang mencatat setiap perubahan pada kode secara terstruktur — siapa yang mengubah, kapan, dan apa isi perubahannya.

Dengan VCS kalian bisa:

  • Melihat riwayat lengkap perubahan sebuah file, kapan pun.
  • Kembali ke versi sebelumnya kalau ada yang rusak.
  • Bekerja paralel dengan banyak orang tanpa saling menimpa pekerjaan.
  • Tahu persis siapa mengubah baris kode mana, dan kenapa.

Version control sendiri berkembang dalam tiga generasi:

  • Local VCS — riwayat cuma tersimpan di satu komputer, tidak bisa dipakai kolaborasi sama sekali.
  • Centralized VCS (contoh: SVN) — satu server pusat menyimpan seluruh riwayat, semua orang terhubung ke situ. Risikonya: kalau server itu down atau rusak, semua orang kena dampaknya sekaligus.
  • Distributed VCS (contoh: Git) — setiap komputer punya salinan riwayat yang lengkap, bukan cuma satu server yang punya.
📌
Git itu terdistribusi

Centralized itu seperti satu buku catatan yang dipegang bendahara kelas — kalau bukunya hilang, semua data kelas ikut hilang. Distributed itu seperti semua anggota kelas punya fotokopi lengkap dari buku itu. Karena terdistribusi, kalian tetap bisa commit, lihat log, dan membuat branch meski sedang tanpa internet.

Kenapa Memakai Git?

Git dibuat tahun 2005 oleh Linus Torvalds (pencipta Linux), karena tim kernel Linux kehilangan akses ke tool berbayar yang biasa mereka pakai untuk mengelola kontribusi dari ribuan developer di seluruh dunia. Linus butuh solusi cepat, dan dalam hitungan hari, Git pun lahir. Sejak itu, Junio Hamano melanjutkan proyek ini sebagai maintainer utama sampai sekarang.

🙂
Kenapa namanya "Git"?

Linus sendiri pernah bercanda bahwa ia selalu menamai proyeknya sesuai dirinya sendiri — dan "git" adalah slang Inggris untuk "orang yang menyebalkan". Analogi yang lebih membantu: Git itu seperti kamera. Setiap kali kalian commit, Git sedang "memotret" kondisi seluruh proyek kalian saat itu juga, dan foto itu tersimpan selamanya.

Sampai sekarang Git menjadi standar industri karena:

  • Cepat — sebagian besar operasi berjalan lokal di komputer kalian.
  • Aman — setiap perubahan diberi checksum (SHA-1/SHA-256), riwayat sulit dimanipulasi diam-diam.
  • Mendukung kerja paralel — lewat branching yang ringan dan cepat.
  • Bisa dipakai offline — commit, lihat riwayat, dan membuat branch tanpa internet.
  • Gratis & open source, dipakai hampir semua perusahaan software di dunia.

Git vs GitHub: Bukan Hal yang Sama

Ini kebingungan paling umum untuk pemula, jadi penting untuk clear sejak awal:

GitGitHub
Perangkat lunak (software) version control Platform/layanan hosting untuk repository Git
Berjalan di komputer kalian, lewat command line Berjalan di server, diakses lewat browser/API
Dibuat oleh Linus Torvalds, open source Dimiliki oleh Microsoft, layanan komersial (ada versi gratis)
Alternatif: tidak ada pengganti langsung, ini standarnya Kompetitor: GitLab, Bitbucket, Gitea

Analogi sederhana: Git itu seperti Microsoft Word — aplikasi untuk membuat dan menyimpan dokumen (kode) beserta riwayat perubahannya. GitHub itu seperti Google Drive — tempat menyimpan dokumen itu di cloud supaya bisa diakses dan dikerjakan bersama orang lain. Kalian bisa pakai Git tanpa GitHub, tapi tidak bisa pakai GitHub tanpa Git.

GitHub Bukan Satu-satunya

GitHub punya beberapa "saudara" yang juga populer, tergantung kebutuhan tim:

  • GitLab — populer untuk DevOps & CI/CD yang terintegrasi.
  • Bitbucket — sering dipakai tim yang sudah pakai produk Atlassian (Jira, Trello).
  • Azure DevOps — umum di ekosistem perusahaan berbasis Microsoft.
  • Gitea & Forgejo — ringan, open source, cocok di-hosting sendiri.

Workshop ini memakai GitHub karena paling populer untuk belajar dan kolaborasi open source — paling banyak tutorial, komunitas, dan lowongan kerja yang menyebutnya.

Konsep Dasar & Alur Kerja

Ada beberapa istilah yang akan terus muncul sepanjang workshop ini:

  • Repository (repo) — folder proyek yang riwayat perubahannya dilacak oleh Git.
  • Commit — "snapshot" atau titik simpan dari kondisi proyek pada waktu tertentu, lengkap dengan pesan penjelas.
  • Working Directory — folder proyek yang sedang kalian edit sehari-hari.
  • Staging Area (Index) — area "ruang tunggu" untuk memilih perubahan mana yang akan disertakan pada commit berikutnya.
  • HEAD — penunjuk ke commit/branch yang sedang aktif saat ini.

Alur kerja dasarnya seperti ini:

alur kerja git 
Working Directory  --git add-->  Staging Area  --git commit-->  Repository (lokal)  --git push-->  GitHub (remote)
   (file diedit)      (dipilih)      (siap disimpan)      (tersimpan permanen)      (dibagikan ke tim)

Jangan khawatir kalau istilah-istilah ini belum terasa nyambung — di Modul 3 kita akan langsung praktik satu per satu sampai alurnya terasa natural.

Poin penting modul ini

Git = alat version control di komputer kalian. GitHub = tempat hosting repo Git di internet supaya bisa berkolaborasi. Keduanya saling melengkapi, bukan pengganti satu sama lain.
