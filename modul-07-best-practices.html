Best Practices

Kebiasaan-kebiasaan kecil ini yang membedakan riwayat Git yang rapi dan bisa dipercaya, dengan yang berantakan dan menyulitkan tim.

Konvensi Pesan Commit: Conventional Commits

Format populer yang dipakai banyak tim & proyek open source adalah Conventional Commits:

format
<tipe>: <deskripsi singkat>

<penjelasan lebih detail, opsional>
TipeDipakai untuk
featMenambah fitur baru
fixMemperbaiki bug
docsPerubahan dokumentasi saja
styleFormat/gaya kode, tanpa mengubah logika (spasi, indentasi)
refactorRestrukturisasi kode tanpa mengubah perilaku
testMenambah/memperbaiki test
choreTugas rutin: update dependency, konfigurasi build, dll
contoh 
feat: tambah validasi email di form pendaftaran
fix: perbaiki tombol submit yang tidak responsif di mobile
docs: update instruksi instalasi di README
chore: perbarui dependency ke versi terbaru

.gitignore Secara Mendalam

File yang tidak boleh ikut dilacak Git, di antaranya:

  • Dependency yang bisa diunduh ulang (node_modules/, vendor/).
  • File environment berisi kredensial (.env).
  • File hasil build (dist/, build/).
  • File spesifik editor/OS (.DS_Store, .vscode/, Thumbs.db).
.gitignore — contoh umum 
# Dependencies
node_modules/
vendor/

# Environment & secrets
.env
.env.local

# Build output
dist/
build/

# OS & editor
.DS_Store
Thumbs.db
.vscode/
.idea/

# Logs
*.log
📌
Sudah terlanjur ter-track?

Menambahkan file ke .gitignore tidak menghapusnya dari riwayat yang sudah ada. Kalau file sudah pernah di-commit, hapus dulu dari tracking dengan git rm --cached nama-file, baru commit lagi.

Atomic Commits: Satu Commit, Satu Tujuan

Usahakan setiap commit hanya berisi satu perubahan logis — jangan mencampur perbaikan bug dengan penambahan fitur baru dalam satu commit. Manfaatnya:

  • Riwayat lebih mudah dibaca dan dipahami.
  • Kalau perlu di-revert, hanya perubahan yang relevan yang terpengaruh.
  • git log jadi dokumentasi hidup tentang evolusi proyek.

Strategi Branching

Ada beberapa pendekatan populer — pilih sesuai skala tim dan kebutuhan rilis:

StrategiCocok untuk
GitHub Flow Tim kecil-menengah, deploy sering (continuous deployment). Hanya main + branch fitur pendek umur. Yang dipelajari di Modul 6 memakai pola ini.
Git Flow Proyek dengan siklus rilis terjadwal (versioned releases). Punya branch develop, release/*, hotfix/* selain main. Lebih formal, cocok tim besar.
Trunk-based Development Tim dengan CI/CD matang. Semua orang commit langsung ke main (branch sangat pendek umur, sering di bawah 1 hari), dilindungi feature flag.

Untuk pemula dan sebagian besar proyek kecil-menengah, GitHub Flow sudah lebih dari cukup.

Jangan Pernah Commit Rahasia (Secrets)

🚫
Kredensial yang sudah ter-commit = anggap bocor

API key, password, token, atau file .env yang pernah ter-commit akan tetap ada di riwayat Git meski filenya kemudian dihapus — dan otomatis terekspos begitu repo di-push ke GitHub (bahkan repo private pun berisiko). Simpan kredensial di .env, pastikan .env masuk .gitignore sebelum commit pertama.

Kalau terlanjur ter-push, langkah minimalnya: cabut/rotasi kredensial tersebut segera (anggap sudah bocor), baru bersihkan riwayat Git (di luar cakupan workshop ini — cari git filter-repo atau BFG Repo-Cleaner).

Checklist Sebelum Push

  • ☐ Jalankan git status — tidak ada file yang tidak sengaja ikut ter-stage.
  • ☐ Jalankan git diff --staged — review ulang isi perubahan.
  • ☐ Pesan commit jelas dan mengikuti konvensi tim.
  • ☐ Tidak ada kredensial/secret yang ikut ter-commit.
  • ☐ Kode sudah dicoba jalan / test (kalau ada) sudah lulus.
  • ☐ Branch sudah sesuai (bukan tidak sengaja commit ke main).
