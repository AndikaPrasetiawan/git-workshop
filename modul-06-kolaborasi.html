Kolaborasi & Pull Request

Ini yang membedakan Git solo dengan Git tim: alur fork → branch → Pull Request → review → merge.

Di modul ini
  1. Fork vs Clone
  2. Alur kolaborasi standar
  3. Membuat Pull Request
  4. Code review
  5. GitHub Issues

Fork vs Clone

Keduanya sama-sama menyalin repo, tapi tujuannya berbeda:

CloneFork
Apa yang terjadi Mengunduh repo ke komputer lokal Membuat salinan penuh repo ke akun GitHub kalian sendiri
Kapan dipakai Kalian punya akses tulis (write access) langsung ke repo, misalnya repo tim internal Kalian tidak punya akses tulis, misalnya berkontribusi ke proyek open source orang lain
Alur clone → branch → push → Pull Request (di repo yang sama) fork → clone hasil fork → branch → push → Pull Request (ke repo asli)

Alur Kolaborasi Standar (GitHub Flow)

Pola ini dipakai hampir semua tim, baik untuk repo tim internal maupun kontribusi open source:

alur kerja
1. (fork jika perlu) → clone repo ke lokal
2. git switch -c fitur/nama-fitur     # buat branch baru dari main
3. ...edit kode, git add, git commit...
4. git push origin fitur/nama-fitur   # push branch ke GitHub
5. Buka Pull Request di GitHub (branch fitur → main)
6. Tim mereview, memberi komentar, meminta perubahan jika perlu
7. Setelah disetujui → merge Pull Request
8. Hapus branch fitur (lokal & remote), lalu ulangi dari langkah 2
⚠️
Jangan commit langsung ke main

Di kerja tim, hindari commit langsung ke branch main. Selalu buat branch baru per fitur/perbaikan — ini memudahkan review dan mencegah kode yang belum teruji masuk ke branch utama.

Membuat Pull Request

  1. Setelah git push branch baru, buka repo di GitHub — biasanya muncul banner kuning "Compare & pull request". Klik itu, atau buka tab Pull requests → New pull request.
  2. Pastikan base adalah branch tujuan (biasanya main) dan compare adalah branch fitur kalian.
  3. Isi judul yang jelas dan deskripsi: apa yang diubah, kenapa, dan cara mengujinya.
  4. Kalau relevan, tautkan ke issue terkait dengan menulis Closes #12 di deskripsi.
  5. Klik Create pull request.

Code Review

Setelah PR dibuat, reviewer bisa:

  • Comment — memberi catatan pada baris kode tertentu tanpa memblokir merge.
  • Approve — menyetujui perubahan, siap untuk di-merge.
  • Request changes — meminta perbaikan sebelum PR bisa di-merge.

Kalau ada revisi, cukup commit & push lagi ke branch yang sama — PR akan otomatis ter-update.

Saat merge, GitHub menawarkan tiga opsi:

OpsiEfek
Create a merge commitMenyimpan seluruh riwayat commit dari branch fitur + membuat satu merge commit baru.
Squash and mergeMenggabungkan semua commit di branch fitur menjadi satu commit rapi di main. Populer untuk menjaga riwayat main tetap bersih.
Rebase and mergeMenambahkan commit dari branch fitur ke atas main satu per satu, tanpa merge commit.

GitHub Issues

Issues adalah tempat melaporkan bug, mengusulkan fitur, atau melacak pekerjaan — sebelum kode ditulis. Buka tab Issues → New issue, beri judul jelas, deskripsi langkah reproduksi (untuk bug), dan label yang sesuai (bug, enhancement, dll).

Hubungkan Issue dengan Pull Request dengan menulis kata kunci seperti Closes #7, Fixes #7, atau Resolves #7 di deskripsi PR — issue akan otomatis tertutup begitu PR di-merge.
